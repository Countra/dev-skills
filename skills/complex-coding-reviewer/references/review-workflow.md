# Review Workflow

## 1. 固定审查问题

开始前确认：

- 用户真正要求什么
- 使用 plan-review 还是 code-review
- 审查目标和基线是什么
- 哪些规范、测试或运行结果与当前目标有关
- 哪些输入只是作者说明，尚未被验证

目标不明确且会改变结论时先补充上下文。不要为了完整感无边界浏览仓库。

## 2. 需求优先

先寻找 missing、extra、misunderstood 三类偏差，再检查工程质量：

- missing：批准行为、错误路径或验收条件未覆盖
- extra：引入未批准行为、依赖、权限或兼容变化
- misunderstood：实现存在但语义、顺序、边界或失败处理错误

实现者的“已修复”“安全”“仅重构”等表述只能作为线索，不能作为结论。

## 3. 风险筛查

根据目标判断是否命中安全权限、隐私、并发完整性、性能资源、API/数据兼容、UI 可访问性或删除依赖。只读取命中的 [risk-playbooks.md](risk-playbooks.md) 章节。

发现超出当前专业能力且可能改变结论的高风险问题时，记录 gap 并建议合格专业审查，不伪装覆盖。

## 4. 证据与严重度

finding 必须绑定当前文件、diff 或计划位置。说明触发条件、可观察影响和最小修复方向；不要把替代设计偏好写成缺陷。

严重度使用 [review-calibration.md](review-calibration.md)。blocking/major 必须在交付前解决或由新证据证明不成立；接受风险或延期不能伪装成 passed。

## 5. 独立上下文

需要 independent review 时创建一个隔离子 Agent，并传递最小、客观、可复现的输入。不得传父代理预判；不得让子 Agent递归派发。

技术失败最多重新创建一次。目标或上下文变化时废弃旧结论，从当前目标重新审查。独立能力不可用且属于安全前提时停止。

## 6. 输出与复审

先输出 findings，之后说明 coverage 和 gaps。没有 blocking/major 时仍说明检查过什么和哪些内容未验证。

Reviewer 不修改目标。调用方修复后重新读取完整目标，确认修复没有引入新回归；不为 minor/advisory 强制循环。
