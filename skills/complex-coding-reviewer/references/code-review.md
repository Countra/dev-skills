# Code Review

## 先核对需求

- 当前 diff 是否覆盖批准目标和错误路径
- 是否存在 missing、extra 或 misunderstood 行为
- 是否修改未批准路径、公共接口、依赖、迁移或权限
- 实现者总结是否与实际代码一致

## 再核对正确性

- 输入边界、空值、顺序、状态转换和错误传播
- 资源释放、取消、超时、重试和重复调用
- 并发、幂等、事务、缓存和持久化一致性
- 平台、编码、路径、时区和环境差异
- 删除、重命名或替换后是否仍有调用方和配置残留

## 工程质量

- 模块职责、依赖方向和抽象是否符合项目现有设计
- 是否引入不必要耦合、重复逻辑或难以验证的隐藏状态
- 日志和错误是否能帮助定位失败，且不泄露秘密
- 新依赖是否与批准选择、manifest 和 lockfile 一致

只有这些问题造成真实缺陷、显著维护风险或验证盲区时才形成 finding。命名和风格偏好交给项目 lint 或 advisory。

## 验证证据

- 测试是否覆盖实际变更和主要失败路径
- 运行结果是否来自当前目标、当前环境和当前代码
- mocked 测试是否遗漏真实集成行为
- 未运行、跳过或仅由作者报告的检查必须写入 gaps

Reviewer 不运行测试。必要验证由 Executor 完成并把结果作为 evidence 提供。

## 定位

finding 使用当前路径和尽可能精确的行号；diff 中的删除问题可定位到调用方、相关保留代码或明确的删除片段。说明触发条件和影响，不只复述代码。
