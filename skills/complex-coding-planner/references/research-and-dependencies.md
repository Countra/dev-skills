# 调研与依赖选择

## 停止条件

调研服务于决策。先读项目规则、源码、测试、配置、manifest、lockfile 和近期提交；这些信息已经能支持方案时停止。

以下情况才使用在线资料：

- 版本、维护状态、平台行为或 API 可能变化
- 新增、替换或升级关键依赖
- 项目事实与记忆中的规范冲突
- 安全、权限、数据迁移、并发或兼容风险仍不明确
- 用户明确要求深入调研或比较方案

稳定事实通常一个官方来源即可。存在候选比较、冲突或重大风险时，再补充独立的一手来源。新证据不再改变选择时结束，不下载或复述整套规范。

## 来源优先级

1. 项目现有规则、实现、测试和 lockfile
2. 语言、框架或平台官方文档与发布记录
3. 官方仓库、标准组织、原始论文或维护者资料
4. 高质量工程资料，仅用于补充实践背景

搜索摘要、榜单和二手文章不能单独支撑高风险结论。

## 依赖选择

先判断是否真的需要新依赖。健康的现有栈、标准库或官方 SDK 能满足需求时优先复用。

新增或替换关键依赖时检查：

- 与项目语言版本、部署目标、许可证和安全边界是否兼容
- 当前稳定版本和发布策略
- 用户规模与同类项目采用情况
- 最近维护、issue/PR 响应和维护者连续性
- 更新时间与采用趋势，而不是单一 stars 或下载量
- API 稳定性、迁移成本、测试能力和退出方案

主流方案明显胜出时不为凑数量制造候选。存在真实权衡时比较 2-3 个方案。最终计划只写选择、关键证据、被否决方案的主要风险和来源链接。

## 规范与架构

先识别项目语言、框架、API 类型和部署环境，再按需查对应官方规范。代码风格、SOLID 和设计模式只有在改变模块边界、错误处理、可测试性或扩展方式时才写入计划。

架构建议必须落到当前仓库的模块、调用关系和验证方式，避免脱离现状的通用清单。
